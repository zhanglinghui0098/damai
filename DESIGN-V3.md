## 3. 核心功能架构 — 双轨设计

### 3.1 模板模式 (零基础) — 主推

**流程** (3 步):
```
选模板 → 填素材 → 出片
   │         │         │
   ▼         ▼         ▼
 89 模板   上传图+文   一键生成
 (按品类   改模板     (调多模型
 分类)     参数)      路由)
```

**页面**:
- `/templates` — 89 个模板网格 (5 分类: 老板出镜/客户家/产品/节日/其他)
- `/templates/[id]` — 单模板详情 + 填素材表单
- `/generate?template=[id]` — 生成中页面 (进度条 + AI 标识)
- `/asset/[id]` — 完成后视频 + "去发布" / "查看创作过程"

**模板元数据** (数据库):
```json
{
  "id": "boss-001",
  "name": "老板出镜讲解",
  "category": "老板出镜",
  "preview": "/templates/boss-001.mp4",
  "script": "你好我是[店主]...今天给你推荐[产品]...",
  "params": [
    { "key": "shop_name", "type": "text", "label": "店铺名" },
    { "key": "product_img", "type": "image", "label": "产品图" },
    { "key": "owner_voice", "type": "audio", "label": "老板声音 (可选)" }
  ],
  "ai_watermark": true,
  "canvas_workflow": "boss-001.json"
}
```

### 3.2 画布模式 (高级, 培训后) — 差异化

**对标**: TapNow 的 Node/Wire/Group/Canvas (已验证 6/18) + LibTV 的"工作流可视化"

**入口**:
- 主页第二排"🎨 画布模式" → /canvas
- 案例卡"查看创作过程" → /canvas?case=v1
- /dashboard "新建工作流" → /canvas?new=1

**页面**:
- `/canvas` — 主画布 (Node/Wire 拖拽)
- `/canvas?case=v1` — 案例模式 (只读, 看别人怎么做的)
- `/canvas?workflow=xxx` — 编辑模式 (编辑自己的工作流)

**节点类型** (5 类):
- 输入: 文本/图片/音频/视频
- 模型: Seedance 2.0 / 即梦 / 可灵 / Vidu (多模型路由)
- 处理: 字幕 / 配音 / 拼接 / 转场
- 输出: 视频 / 图片 / 文档
- 控制: 条件分支 / 循环 / 错误处理

**关键设计**: 每个模板对应一个画布工作流 JSON, **零基础模式 = 工作流已配好**; **画布模式 = 自己拖节点配工作流**

### 3.3 双轨关系图

```
                    ┌─────────────────┐
                    │ 89 个行业模板    │ ← 我们预制 (官方)
                    │ (含 canvas JSON) │

                             │
              ┌──────────────┴──────────────┐
              ▼                             ▼
    ┌──────────────────┐         ┌──────────────────┐
    │ 模板模式 (零基础) │         │ 画布模式 (高级)   │
    │ 选模板 + 填素材   │         │ 自己拖节点配工作流 │
    │ 一键出片         │         │ 灵活 + 自由组合   │
    │ 隐藏工作流细节    │         │ 适合批量 + 个性化  │
    └──────────────────┘         └──────────────────┘
              │                             │
              └──────────────┬──────────────┘
                             ▼
                    ┌─────────────────┐
                    │ 多模型路由层     │ ← Seedance 2.0 主用
                    │ 统一 AI 标识     │ ← 强制水印
                    └─────────────────┘
```

## 5. 对话框接入 MiniMax — 4 方案对比 (关键决策)

### 5.1 背景信息 (已实测)

- MiniMax API 端点: `https://api.minimax.chat/v1` (从 config.yaml 确认)
- 协议: OpenAI 兼容
- API key: 已配 `MINIMAX_CN_API_KEY`
- 当前模型: `MiniMax-M3`

### 5.2 4 方案对比

| 方案 | 描述 | 优点 | 缺点 | 成本 | 我的建议 |
|---|---|---|---|---|---|
| **A. 飞书原生聊天** | 用户在飞书跟 Hermes 聊 (现在就是这样) | 零成本, 已跑通 | 跟主页"对话框"分离, 跳转 | ¥0 | 不推荐作为主页方案 |
| **B. web 嵌 chat UI → 直调 MiniMax API** | 主页有 chat 框 → 后端 proxy → MiniMax | UI 一体化, 体验最好 | 需 API key 安全保护 + 后端代理 | ¥0 (用现 key) | **推荐 (中期)** |
| **C. web 嵌 chat UI → 转发到飞书 Hermes** | web chat → 飞书 webhook → Hermes 回应 | 复用飞书通道 | 延迟 5-10s, web 端 polling 复杂 | ¥0 | 不推荐 |
| **D. 主页对话框 = "AI 引导" + MiniMax 真正 LLM** | web chat 调 MiniMax → 输出结构化引导 → 跳模板 | 既有 LLM 智能化, 又不需暴露 key | 需写后端 proxy | ¥0 | **推荐 (落地路径)** |

### 5.3 推荐路径: B/D 混合 (分阶段)

**Phase 1 (1 周, 落地)**: D — 主页对话框 → 后端 proxy → MiniMax M3
- 后端: Next.js API route `/api/chat`, 调 MiniMax M3
- 前端: chat 框 + 流式输出 (Vercel AI SDK)
- MiniMax prompt: "你是大脉助手, 引导用户选模板/画布, 输出结构化建议"
- **关键: 不暴露 API key 给前端, 全部走后端**

**Phase 2 (2 周, 升级)**: B — 加工具调用 (function calling)
- MiniMax M3 调起工具: `recommend_template()`, `start_canvas()`, `show_case_studies()`
- 用户问"老板出镜怎么做" → AI 直接推荐模板 + 跳 /templates/boss-001

**Phase 3 (备案后)**: C — 接多模型路由
- 不同任务用不同模型 (写脚本用 MiniMax, 生图用即梦, 生视频用 Seedance 2.0)
- MiniMax 当"调度员"

我没 100% 验证的事:
1. **MiniMax 是否支持 function calling** — 需要看官方文档 (没搜到, **待确认**)
2. **MiniMax 是否有"agent"产品** (类似 OpenAI Assistants) — **不确定, 待查**

## 6. Codex 协作改进方案

### 6.1 现状痛点 (用户没说, 我观察到)

- 现有 brief `damai-codex-brief-overview.md` 17KB 偏厚 → 沙盒 agent 看不完
- 用户手动复制粘贴 2 次 (给 Codex + 回 NAS) → 痛
- Codex 不知道"现在哪个 task" → brief 要写明状态
- Codex 设计 vs 写代码 角色没分开 → 容易混

### 6.2 改进方案: Brief 拆分 + 角色分工

#### 6.2.1 Brief 文件拆分 (从 1 个变 4 类)

```
codex-deliveries/
├── README.md                                  ← 使用说明
├── damai-codex-brief-overview.md     (8KB)    ← 项目全貌 (简化版)
├── damai-codex-brief-current.md      (3KB)    ← 当前 in-progress task 状态
├── damai-codex-brief-design-{topic}.md (5-8KB) ← 设计任务 (例: 设计主页 v3)
└── damai-codex-brief-task{NN}.md     (5-8KB)  ← 写代码任务
```

**好处**:
- 用户按需复制 (10KB 比 17KB 好)
- Codex 看 brief 知道自己扮演什么角色 (设计 vs 写代码)
- 状态更新不需要重写 overview

#### 6.2.2 角色分工

| Codex 角色 | 触发 | 产出 | 谁验证 |
|---|---|---|---|
| **设计师** | 用户说"让 Codex 出个方案 A/B" | design brief → 设计稿 (markdown + ASCII) | 用户 + Hermes 评估, 选 A 实施 |
| **码农** | 用户说"让 Codex 写 X 模块" | task brief → 代码 (diff + 测试) | Hermes 跑测试 + 评估 |
| **审计员** | 用户说"让 Codex 看看有没有问题" | audit brief → 风险清单 | Hermes 复核 |

#### 6.2.3 简化中转流程 (3 步)

```
[1] 用户说 "让 Codex 想想 X"
       ↓
[2] Hermes 生成 brief 文件 (1 步, 自动化) → 写到 SMB
       ↓
[3] 用户打开 SMB → 复制 → 贴给 Codex (1 步)
       ↓
[4] Codex 输出 → 用户复制 → 贴回 SMB (1 步)
       ↓
[5] Hermes 读 + 评估 + 报告
```

**总中转成本 = 复制粘贴 2 次 (跟之前一样), 但 brief 质量更高**

### 6.3 设计 v3 的具体协作 (举例)

如果用户要 Codex 帮忙出**主页 v3 设计方案**:

1. **Hermes 生成 brief**:
   ```
   /opt/data/projects/damai/codex-deliveries/damai-codex-brief-design-homepage-v3.md (5-8KB)
   ```
   内容: 用户原话 + DESIGN-V3.md §2 草案 + 让 Codex 出 2 个备选 (风格 A vs 风格 B)

2. **用户复制 → 贴给 Codex**:
   ```
   看上面这个 brief, 给我 2 个主页设计备选 (风格 A 苹果极简 / 风格 B 品牌特色)
   每个备选用 ASCII 草图 + 关键文案 + 推荐理由
   ```

3. **Codex 输出**:
   - 风格 A 完整草图
   - 风格 B 完整草图
   - 我 (Hermes) 的推荐

4. **用户 + Hermes 评估 → 选 A/B → 让 Codex 实现或 Hermes 实现**
